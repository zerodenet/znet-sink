# 交互模式约束

本文档定义 GUI 的简约模式与专业模式。模式是应用级交互策略，不是前端菜单样式，也不是内核能力开关。

## 核心原则

- GUI 只对接自研内核 `zero`；短期接入其他内核不作为当前产品目标。
- 简约模式是收起复杂度，不是降级能力。
- 专业模式暴露 GUI 能承接的内核全能力。
- 模式语义由 Rust 后端定义，前端只消费 Rust 计算后的交互面。
- 前端隐藏入口不是权限边界；权限边界必须在 Rust 命令层。
- 模式切换不得删除、禁用、改写已有专业配置。

## 分层顺序

模式相关逻辑必须按下面顺序流动：

1. 自研内核声明事实能力，例如规则、规则集、策略组、连接表、DNS、TUN、脚本、MITM。
2. Rust 后端读取应用配置中的 `uiMode`，并解释模式语义。
3. Rust 后端根据内核能力、当前配置、当前模式、运行状态计算有效交互面。
4. 前端根据有效交互面渲染菜单、页面、按钮、禁用态和提示。
5. 用户操作回到 Rust 命令层时，Rust 再次校验当前模式是否允许该操作。

前端不得自行解析内核配置来判断高级能力是否显示，也不得只靠路由、菜单或组件隐藏来限制操作。

## 模式定义

### 简约模式

简约模式面向只需要完成基础连接的用户。它保留内核运行能力，但收起高级概念和高级操作。

简约模式应展示和允许：

- 启动、停止、运行状态。
- 系统代理开关。
- 当前节点或策略选择。
- 订阅同步。
- 基础错误提示和必要日志。
- 基础应用设置。

简约模式不应展示，也不应允许操作：

- 规则、规则集、路由分流配置。
- 内核配置快照、启动参数、高级导入导出。
- IPC 调试、能力诊断、运行时原始 JSON。
- DNS、TUN、脚本、MITM 等高级细项配置。
- 连接表中的高级控制操作，例如强制关闭指定连接；基础只读展示可按产品需要另行开放。

如果专业模式下已经配置了规则、DNS、TUN 等高级能力，切换到简约模式后，这些配置仍然可以被内核使用；GUI 只是隐藏并禁止编辑这些复杂度。

### 专业模式

专业模式面向需要完整控制和诊断的人。它应展示 GUI 能承接的自研内核全能力。

专业模式应展示和允许：

- 规则、规则集、路由分流。
- 策略组、节点测速、手动选择、自动选择诊断。
- 连接列表、连接详情、连接控制。
- DNS、TUN、脚本、MITM 等内核声明支持的高级能力。
- 内核配置导入导出、配置快照、启动参数、IPC 调试。
- 健康状态、运行时统计、详细日志和必要诊断信息。

专业模式仍要受真实内核能力约束。内核没有声明支持的能力，即使在专业模式也不能伪造入口。

## 有效交互面

Rust 后端应提供一个面向前端的快照，描述当前模式下每个功能的交互状态。该职责由 `gui_interaction_surface_snapshot` 承担。

推荐每个功能项至少包含：

```json
{
  "key": "ruleSets",
  "visible": false,
  "operable": false,
  "readonly": false,
  "reason": "hidden in lite mode"
}
```

字段语义：

| 字段 | 说明 |
| --- | --- |
| `key` | 稳定功能标识 |
| `visible` | 当前模式下是否可见 |
| `operable` | 当前模式下是否可执行写入或控制操作 |
| `readonly` | 可见但只读 |
| `reason` | 不可见、不可操作或只读的原因 |

`enabled` 只能表示能力是否成立；当需要表达模式约束时，应优先拆成 `visible` 与 `operable`，避免把“内核不支持”和“简约模式收起”混在一起。

## 命令层约束

Rust 命令层必须作为最终保护边界。简约模式下，专业能力的写命令必须拒绝执行。

推荐策略：

| 命令类型 | 简约模式行为 | 专业模式行为 |
| --- | --- | --- |
| 基础状态查询 | 允许 | 允许 |
| 启动/停止内核 | 允许 | 允许 |
| 系统代理开关 | 允许 | 允许 |
| 订阅同步 | 允许 | 允许 |
| 策略选择 | 允许 | 允许 |
| 规则/规则集写入 | 拒绝 | 按能力和参数校验 |
| 高级内核配置写入 | 拒绝 | 按能力和参数校验 |
| IPC 调试命令 | 拒绝 | 允许 |
| 高级诊断查询 | 拒绝或隐藏 | 允许 |

被模式拒绝的命令应返回稳定错误码。推荐使用 `mode_restricted`：

```json
{
  "code": "mode_restricted",
  "message": "rule set management is only available in pro mode",
  "details": {
    "uiMode": "lite",
    "feature": "ruleSets"
  }
}
```

## 数据不变量

- `uiMode` 只改变 GUI 的交互面，不改变内核事实能力。
- `uiMode` 切换不删除任何代理配置、规则集、订阅、内核配置或日志。
- 简约模式下隐藏的专业配置必须保留，并可继续参与内核运行。
- 用户偏好 `hiddenMenuKeys` 只表示用户手动隐藏菜单，不得承载业务能力或模式策略。
- 能力快照和交互面应由 Rust 侧生成，前端只负责展示。
- 前端可以做体验层兜底，但不能成为模式限制的唯一实现。

## 功能归属

| 功能 | 简约模式 | 专业模式 |
| --- | --- | --- |
| 总览状态 | 可见、可操作 | 可见、可操作 |
| 启动/停止内核 | 可见、可操作 | 可见、可操作 |
| 系统代理 | 可见、可操作 | 可见、可操作 |
| 节点或策略选择 | 可见、可操作 | 可见、可操作 |
| 订阅同步 | 可见、可操作 | 可见、可操作 |
| 基础日志 | 可见 | 可见 |
| 规则与规则集 | 不可见、不可操作 | 按能力可见、可操作 |
| 连接高级控制 | 默认不可见、不可操作 | 按能力可见、可操作 |
| 内核配置快照 | 不可见、不可操作 | 可见、可操作 |
| IPC 调试 | 不可见、不可操作 | 可见、可操作 |
| 能力诊断 | 不可见 | 不作为菜单入口，仅内部诊断 |
| DNS/TUN 高级设置 | 不可见、不可操作 | 按能力可见、可操作 |
| 脚本/MITM | 不可见、不可操作 | 按能力可见、可操作 |

## 实现检查清单

涉及模式逻辑的改动必须检查：

- Rust `AppConfig.ui.uiMode` 是否是模式唯一来源。
- Rust 能力快照或交互面是否表达了当前模式的可见性和可操作性。
- 专业能力的写命令是否有 Rust 侧 guard。
- 前端是否只消费 Rust 给出的交互面，没有自行复制模式规则。
- 模式切换是否只影响 UI 和命令准入，没有修改业务数据。
- 新增功能是否明确归入简约模式、专业模式或两者共有。
